# Upstream Integration

This document makes the upstream connection story more concrete.

## Recommended upstream model

The recommended upstream is **an extracted bookmark DB / export JSON** that is maintained separately from the viewer layer.

In practice, the public repo assumes this upstream can already provide a private-style JSON export.

---

## Minimum contract

The practical contract is:

1. upstream system produces a bookmark export JSON on a stable path
2. `refresh_bundle.py` reads that path
3. the bundle output is refreshed in place
4. optional scheduler reruns the same command later

That means the public repo does **not** need to own:

- browser scraping logic
- account/session handling
- long-lived private metadata storage

It only needs a stable handoff file.

---

## Why this is important

This separation improves both trust and maintainability.

### Upstream side stays free to be:

- private
- richer in metadata
- operationally messy if needed
- specific to one user's extraction stack

### Public bundle side stays:

- smaller
- cleaner
- shareable
- schedulable
- easier to explain to new users

---

## What to automate first

If you want a realistic first automation target, do this first:

- keep one stable upstream JSON path
- run `refresh_bundle.py` against it
- publish or inspect the resulting bundle

That gives users an end-to-end path without requiring them to adopt your full private extraction stack.

---

## Distribution recommendation

If you distribute this project, recommend the upstream source in this order:

1. the user's own extracted bookmark DB export
2. a private-style export JSON generated by another tool
3. the bundled sample data only for evaluation/demo

The sample should never be framed as the real upstream story.
